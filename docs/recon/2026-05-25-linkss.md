# Recon: linkss screensaver hack — 2026-05-25

Investigated as M4.1 "sleep / always-on path A". Goal: keep the latest
dashboard visible on the panel even after the device sleeps.

## What's installed

The Kindle has NiLuJe's **screensavers hack**, `linkss` v0.25.N (see
`/mnt/us/linkss/info.txt`). It was already present when we started the
project — not something we installed.

Reference: <http://www.mobileread.com/forums/showthread.php?t=195474>

## How it works

The hack publishes an FSP (FUSE) bind mount so that whatever the
framework reads from its expected screensaver path is actually served
from a user-writable directory under `/mnt/us`:

```
$ mount | grep screensaver
fsp on /usr/share/blanket/screensaver type fuse.fsp (rw,nosuid,nodev,noatime,...)
```

- Framework-visible path: `/usr/share/blanket/screensaver/bg_ss00.png`
- Backing file (what we actually write): `/mnt/us/linkss/screensavers/bg_ss00.png`
- Perms on the backing file are `0777` — no `mntroot rw` dance required.

When `powerd` puts the device into screensaver, the framework reads
`bg_ss00.png` via the fuse mount and renders it to the panel. Whatever
we drop into that path becomes the next screensaver image, full-screen,
600×800.

Other observations from this Kindle:

- `/mnt/us/linkss/bin/linkss` is the management shell script.
- `/mnt/us/linkss/screensavers/` contains exactly one PNG today
  (the default 2021 placeholder).
- `lipc-get-prop com.lab126.powerd status` reports normal state
  transitions; nothing is currently forcing the device awake.

## Implication for the dashboard

We can keep the existing `eips -g` foreground path *and* also copy each
freshly-fetched PNG to `bg_ss00.png`. Result:

- **Device awake:** `eips` draws the new image immediately (unchanged).
- **Device entering screensaver:** framework draws our latest PNG —
  the dashboard remains visible.
- **Device in deep sleep:** the panel is already holding the last
  rendered frame (eink), so nothing additional is needed there. The
  problem we hit during M2 wasn't the *image* going away — it was Wi-Fi
  dropping so the *next* refresh couldn't fetch. linkss doesn't fix
  that; it does make the wait-for-wake period look intentional.

## Tradeoffs vs path B (`preventScreenSaver`)

Path A wins on simplicity and battery — the device sleeps normally,
saving power, and the screensaver pipeline does what it's already
designed to do. Path B (`lipc-set-prop preventScreenSaver 1`) would
keep cron firing more reliably across the day at the cost of battery
and an extra moving part to maintain.

Decision: do path A first; revisit B only if we observe long
no-refresh windows in real-world use.

## Outcome (later the same day)

Path A alone proved insufficient: empirical test showed cron was suspended
along with the kernel, so refreshes only happened when a user tapped the
device. Path B (`preventScreenSaver` always-on) was then tested and worked
but rejected on battery grounds.

The final architecture we landed on is **sleep + scheduled wake** — see
[2026-05-25-wake-investigation](2026-05-25-wake-investigation.md) and
[D14](../decisions.md). With the daemon running, the stock framework is
stopped and this screensaver hack is no longer reached; the `bg_ss00.png`
publish in `refresh.sh` is kept as a safety net for the case where the
daemon is off and stock UI is back.
